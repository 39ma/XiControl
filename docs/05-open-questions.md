# Открытые вопросы — проверить на реальной машине ДО написания кода

Всё это выполняется в PowerShell от администратора на целевом ноутбуке.
Пробы с `operation=0xFA` (GET) — **только читают**, безопасны. Пробы с `0xFB` (SET) — меняют состояние, осторожно.

---

## 0. Хелпер для вызова метода

Сначала посмотреть сигнатуру метода:
```powershell
(Get-CimClass -Namespace root/wmi -ClassName MiCommonInterface).CimClassMethods
(Get-CimClass -Namespace root/wmi -ClassName MiCommonInterface).CimClassMethods['MiInterface'].Parameters
```
Это покажет точные имена параметров (`InData`/`OutData`) и их типы — нужно, чтобы правильно собрать вызов.

---

## Вопрос 1. Права: нужен ли SYSTEM для SET? 🔑

Определяет, можно ли обойтись без службы.

- [x] Установлено: **даже GET требует администратора** (без elevation `Get-CimInstance` → «нет доступа к CIM»). Значит приложению нужен `requireAdministrator`.
- [x] SET из обычного admin-процесса **работает** (включение защиты 0x02→1 прошло). → **Служба ради привилегий НЕ нужна.** Вопрос закрыт.
- [ ] Проверить: команда режима производительности реально сработала (вентиляторы/поведение) или вернула отказ.
- **Если admin достаточно** → служба не нужна (план А).
- **Если нужен SYSTEM** → минимальная служба для Set (план Б).

## Вопрос 2. Заряд: точный порог и «слетание» 🔑

- [x] Статус читается: `GET 0x10 sub=0x02` → `OUT[6]`.
- [x] ✅ **WMI-заряд подтверждён:** `0x02`=1 → стоп ≈80%, `0x02`=0 → заряд до 100% (доказано переключением, повторяемо). Держит WMI, не Windows.
- [x] Порог **фиксирован ≈80%** (не 70). Произвольный % недоступен. Модель для приложения = **бинарно 80/100**.
- [x] EC сбрасывает защиту при смене питания/сне → **реализован ChargeGuard** (`SystemEvents.PowerModeChanged`, переарм `0x02` с дебаунсом). ✅ Проверено на железе — держит надёжно.

## Вопрос 3. События: ловятся ли из user-сессии? 🔑

- [ ] Подписаться на `HID_EVENT20` и нажать Fn-клавишу (напр. Fn+F4 mic):
```powershell
Register-CimIndicationEvent -Namespace root/wmi -Query "SELECT * FROM HID_EVENT20" -SourceIdentifier mifs
# нажать Fn-клавишу, затем:
Get-Event -SourceIdentifier mifs | ForEach-Object { $_.SourceEventArgs.NewEvent.EventDetail }
```
- [x] ✅ **События ловятся из сессии.** Формат `[0]=01 тип, [1]=код, [2]=значение`. Коды: mic `0x21`, подсветка `0x05`, + `0x01/0x1B/0x23/0x24/0x25/0x26` (ярлыки уточнить по одной клавише). → **план А, служба не нужна.**

## 🏁 ИТОГ 05 — всё ключевое закрыто
- Права: admin хватает, **служба не нужна**.
- Заряд: **80/100 через WMI** (`0x10` sub `0x02`), подтверждён вживую.
- Режимы: **Quiet/Turbo/Full-speed/Auto** (`0x08`), Balance нет.
- События: **ловятся из сессии** (`HID_EVENT20`).
- Телеметрия: отсутствует на TM2424.
- Архитектура: **одно трей-приложение** (C#/.NET 8/WinForms, admin, автозапуск).
- Остаточное — всё закрыто позже: ярлыки клавиш → [07-keymap.md](07-keymap.md);
  ChargeGuard реализован (v0.1); `0x0C` — рудиментный слот TPLock, см. [01-wmi-protocol.md](01-wmi-protocol.md).

## Вопрос 4. Режимы производительности на этой модели

- [x] Модель: SKU = `TM2424-77079` → префикс **`TM24`** ✅ → режим **Auto (0x09) доступен**.
- [x] Текущий режим читается: GET `0x08` → `OUT[4]` (был Auto 0x09).
- [x] SET работает: приняты **Quiet(0x02) / Turbo(0x03) / Full-speed(0x04) / Auto(0x09)**; Balance(0x01) отклонён.
- [x] Позже (v0.3) найден пятый — **Eco(0x0A)**, скрытый: гасит подсветку клавиатуры, снижает яркость. → 5 режимов в UI.
- [ ] (опц.) Full-speed на USB-C vs DC — поведение (не критично для реализации).

## Вопрос 5. Телеметрия и доп-функции (GET, безопасно)

Перебрать чтением `operation=0xFA`, посмотреть осмысленность ответа:
- [ ] `cmd 0x0D` — обороты вентиляторов (CPU `[0..1]`, GPU `[2..3]`, SYS `[6..7]` LE).
- [x] `cmd 0x16` — температура CPU через MIFS: `0xE0` (не поддерживается). **Но температуры решены
  driver-free мимо MIFS** — через Intel DPTF: WMI `root\wmi\EsifDeviceInformation`, `.Temperature`
  (°C) по доменам; берём максимум активных. Проверено на TM2424 (в бенче ~90 °C). Реализовано в «Мониторе».
- [ ] `cmd 0x13` — тип БП (1=USB-C, 2=DC).
- [ ] `cmd 0x12` — яркость подсветки клавиатуры (GET), затем аккуратно SET 0..10 / 128.
- [ ] `cmd 0x09` — режим GPU (GET). SET **не трогать** пока не поймём последствия (риск потерять вывод).

## Вопрос 6. Общая безопасность перебора

- [ ] Перебор неизвестных `cmd` делать **только с `operation=0xFA` (GET)**.
- [ ] SET на неизвестные функции не слать — можно записать в прошивку что-то нежелательное.
- [ ] Вести журнал: `cmd`, отправленный буфер, `OutData` — сложить в `../reference/probe-log.md`.

## Вопрос 7. TDP / Power Limit (PL1/PL2) — можно ли задать ватты? ✅ ЗАКРЫТ (нельзя)

**Итог (2026-07-13): driver-free ползунок TDP в ваттах на TM2424 невозможен.** Три схождения:
1. `probe-cpu-power.ps1` → `0xE0` на всех `0x16–0x18`, поверх полного свипа `0x01..0x30`: **команды мощности в MIFS нет.**
2. Официальный Xiaomi PC Manager требует **китайской Windows** (на обычной не работает) — её нет и разворачивать ради этого не будем → референс на практике не протрейсить, скрытую команду негде подсмотреть. Linux-драйвер: слот «CPU power» — для игровой линейки Tongfang, на нашей тонкой Bitland = `0xE0`.
3. Реальные PL на платформе держит **Intel DTT (ACPI/DPTF)** — драйвер Intel, не наш WMI-канал; user-mode API для произвольных ватт нет. Единственный доступ — запись RAPL MSR = kernel-драйвер (WinRing0/XTU) = **против принципа проекта**.

Замена: 5 дискретных режимов (`0x08`) — это и есть driver-free пресеты мощности прошивки. Если нужна granularity тоньше — опц. `powercfg` «максимум CPU %» (процент частоты, **не ватты**, с оговорками на гибридных Intel).

<details><summary>Исходный разбор вопроса (до закрытия)</summary>

Контекст: запрос на ползунок «конкретный TDP CPU+iGPU» (ресерч 2026-07-13). Полный разбор — [[tdp-power-limit-feasibility]] в памяти.

**Две принципиально разные развилки:**
- **MSR/RAPL** (`MSR 0x610`, PL1/PL2) — это путь «драйвера Intel» / XTU / ThrottleStop. Запись требует Ring 0 → **kernel-драйвер → против принципа проекта** (см. [03-architecture.md](03-architecture.md)). Не рассматриваем.
- **Через прошивку (ACPI/WMI)** — единственный driver-free путь: попросить firmware выставить PL, как g-helper делает у ASUS (эндпоинты SPL/sPPT/fPPT ≈ PL1/PL2, одна ACPI-запись). Работает, **только если MIFS экспонирует такую команду.**

iGPU на Intel питается из **общего пакетного бюджета** мощности вместе с CPU — отдельного «ползунка ватт для iGPU» в user-mode не существует; пакетный PL режет CPU+iGPU вместе.

**Уже известное (частично отрицательное):**
- [x] Полный GET-свип `0x01..0x30` на TM2424 (см. [01](01-wmi-protocol.md) §2): **`0x17` (23 = «CPU power» в игровой таблице Tongfang) отвечает `0xE0` — не поддерживается.** Также `0x14/0x15` (MAX_FAN) = `0xE0`. → у тонкой линейки Bitland слота питания в WMI, похоже, нет.
- ⇒ предварительный вывод: настоящий ватт-лимит через MIFS на TM2424 **скорее всего недоступен**, но свип был GET-only по sub `0x00–0x08` — не исчерпывающий.

**Что проверить на машине (есть TM2424 со всеми драйверами вендора):**
- [ ] Пере-снять GET `0x16..0x18` с **полным дампом `OutData`** (не только `OUT[1]`), по нескольким `arg` — вдруг статус зависит от суб-функции. → `reference/probe-cpu-power.ps1` (GET-only, безопасно).
- [ ] **Главное — трейс официального софта.** С установленными драйверами вендора (Xiaomi PC Manager / Intel DTT) снять WMI/ETW-трейс во время смены режимов и посмотреть, **шлёт ли официальная утилита вообще какой-либо PL-related вызов**:
  - WMI-вызовы `MiInterface`: `Get-WinEvent -LogName Microsoft-Windows-WMI-Activity/Operational` (или `wpr -start` / WPA на провайдер WMI-Activity) — ищем новые `cmd` в 32-байтном буфере помимо `0x08`.
  - Если режимы = только `0x08` + политики Intel DTT (ACPI, не MIFS), значит ватт-команды в нашем канале нет.
- [ ] Что делает **Intel DTT** (`Intel(R) Dynamic Tuning`) на сменах профиля — не он ли выставляет PL через ACPI. Это **не** MIFS (драйвер вендорский), но объясняет, «где» на этой платформе живут ватты.
- [ ] **SET неизвестных `cmd` — НЕ слать вслепую** (правило В6). Только если трейс покажет конкретный безопасный `cmd`+`arg`.

**Развилка результата:**
- Трейс нашёл MIFS-команду PL → **делаем driver-free ползунок / пресеты по ваттам.** 🎉
- PL живёт только в Intel DTT/ACPI, не в MIFS → через наш канал ватты недоступны. Честный вывод: оставляем дискретные пресеты (`0x08`), опц. `powercfg` «максимум CPU %» (это **процент частоты, не ватты**). Вопрос закрыть с этим выводом.

> Как закрылось (2026-07-13): проба дала `0xE0`; официалку протрейсить нельзя — она под китайскую Windows и не ставится, режимы переключались нашим приложением. Реализовалась вторая ветка развилки. Итог — вверху секции.
