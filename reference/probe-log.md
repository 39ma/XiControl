# Журнал проб MIFS WMI

## Машина

| Поле | Значение |
|------|----------|
| Host | DESKTOP-PTM7TOU |
| Производитель / модель | XIAOMI Xiaomi Book Pro 14 |
| SKU | `TM2424-77079` → префикс **`TM24`** = семейство 2025 |
| Инстанс | `ACPI\PNP0C14\MIFS_0` |

## Подтверждено

- ✅ Класс `MiCommonInterface` / метод **`MiInterface`** существует.
- ✅ Сигнатура: `MiInterface(InData: UInt8[] [in], OutData: UInt8[] [out], ReturnCode: UInt16 [out])`.
- ✅ **WMI требует администратора** (даже GET). Приложению нужен `requireAdministrator`.
- ✅ `ReturnCode` = 0 у всех вызовов (это статус COM-метода, не прошивки).

## Формат ОТВЕТА (OutData) — расшифрован

```
OUT[0] = 0x00
OUT[1] = СТАТУС ПРОШИВКИ:  0x80 = OK/поддерживается,  0xE0 = не поддерживается
OUT[2] = 0x00
OUT[3] = эхо cmd (только если поддерживается)
OUT[4] = значение / эхо arg   (для perf mode здесь текущий режим)
OUT[6] = значение для charge-protect (статус вкл/выкл)
```

## Поддерживаемые команды на TM2424 (arg=0, GET)

| cmd | статус | OUT | Значение |
|-----|--------|-----|----------|
| `0x08` Perf mode | `0x80` ✅ | `00 80 00 08 09 …` | режим `OUT[4]` = **0x09 (Auto)** — сейчас активен |
| `0x0A` Микрофон | `0x80` ✅ | `00 80 00 0A 00 …` | ⚠️ `OUT[4]` **всегда 0** независимо от mute (проверено вкл/выкл лампой) → GET(arg=0) НЕ читает состояние. Вероятно write-only / состояние только через события (`HID_EVENT` sub-cmd 0x21) или нужен sub-arg (напр. 0x05). Свип покажет. Фича необязательная. |
| `0x0C` **???** | `0x80` ✅ | `00 80 00 0C 00 …` | неизвестная функция, значение 0 — **надо разобраться** |
| `0x10` Charge protect | `0x80` ✅ | `00 80 00 10 02 … OUT[6]=00` | sub-func `0x02`, статус `OUT[6]` = **0 (выкл / 100%)** |

## НЕ поддерживаются (вернули `0xE0` при arg=0)

`0x09` GPU, `0x0D` вентиляторы, `0x12` подсветка, `0x13` тип БП, `0x16` темп CPU — и весь остальной диапазон `0x00–0x2A`, кроме перечисленных выше.

> ⚠️ Возможно, части из них нужен sub-arg (как `0x02` у заряда). Проверяем свипом — `probe-readonly-2.ps1`.
> Если и со свипом `0xE0` — значит на этой модели функции физически нет (телеметрия из Linux-драйвера — для других Bitland-моделей).

## Открытые вопросы после 1-го прогона

1. **cmd `0x0C`** — что это? (свип sub-arg + осторожный анализ)
2. **Телеметрия** — оживает ли с sub-arg? (свип)
3. **SET** — хватает ли admin (сейчас не проверить без записи). Ждёт `probe-set.ps1` + твоё «го».
4. **События** `HID_EVENT20` — ловятся ли из сессии (Вопрос 3).

## Прогон 2 — свип sub-аргументов (`probe-readonly-2.ps1`)

### 🔑 Заряд `0x10` — под-функции (arg = селектор под-функции)

| sub-arg | статус | OUT[6] | Гипотеза |
|---------|--------|--------|----------|
| `0x00` | `0x00` | — | невалидно |
| `0x01` | `0x80` | `0x64` = **100** | **ЛИМИТ ЗАРЯДА В %** (сейчас 100). Читается — писать? → SET-проба |
| `0x02` | `0x80` | `0x00` | вкл/выкл (0=выкл). Это использует MIControl |
| `0x03` | `0x80` | `0x01` | флаг (AC / charging / capability?) |
| `0x04`–`0x06` | `0x80` | `0x00` | ? |
| `0x07`–`0x08` | `0x00` | — | невалидно |

➡️ **Возможно, произвольный % заряда доступен через WMI** (под-функция 0x01) — это отменило бы вывод «только 70/100». Проверить `probe-set.ps1`.

### Прочее

| cmd | Находка |
|-----|---------|
| `0x08` perf | GET игнорирует arg — всегда режим в `OUT[4]` (0x09 Auto). arg нужен только для SET. |
| `0x0A` mic | sub-func **`0x05`** → `OUT[6]=0x01`. Состояние читается тут, не через arg=0. Совпадает с `Put(0x0A,0x05,…)` MIControl. |
| `0x0C` ??? | sub-func **`0x02`** → `OUT[6]=0x01` (включённый тумблер). Назначение неизвестно. |
| Телеметрия `0x0D/0x12/0x13/0x16/0x09`, а также `0x0B/0x0E/0x0F` | `0xE0` на **всех** sub-arg → на TM2424 отсутствуют. Окончательно. |

### Вывод для проекта
- Доступные функции на этой модели: **режим (`0x08`)**, **заряд (`0x10`, с под-функциями)**, микрофон (`0x0A`), загадочный `0x0C`.
- Телеметрии (вентиляторы/темп/GPU/подсветка через 0x12) **нет** → соответствующие «Nice»-фичи из каталога вычёркиваем.
- **Открыт вопрос переменного %** — решает SET-проба под-функции 0x01.

## Прогон 3 — SET-проба (`probe-set.ps1`)

| Шаг | Результат |
|-----|-----------|
| Исходно | лимит(0x01)=100, защита(0x02)=0 |
| Включить защиту (0x02→1) | ✅ прочиталось 1 — **SET из обычного admin работает** |
| Лимит после включения | всё ещё 100 (не упал в 70) |
| Запись 80 в 0x01 | ❌ прочиталось 100 — **запись не принята** |
| Восстановление | лимит 100, защита 0 — ОК |

### Выводы
- ✅ **Вопрос 1 закрыт:** обычного администратора достаточно для SET. **Служба ради привилегий не нужна.**
- ⚠️ Под-функция `0x01` через `{in[4]=0x01, in[6]=value}` при включённой защите **не записывается** и не отражает 70. Вероятно: read-only, либо порог задаётся иначе, либо писать нужно при ВЫКЛ защите.
- 🔬 **Решает эксперимент «в бою»** (`charge.ps1 -Limit 60` → подключить БП → смотреть, где встанет заряд). Регистр неоднозначен — реальная отсечка даст факт.

### Гипотеза модели (пока)
Скорее всего **бинарно 70/100**: `0x02` = вкл/выкл, порог фиксирован прошивкой (~70), `0x01` = setpoint по умолчанию 100 (read-only или пишется только при выкл). Подтвердить/опровергнуть зарядкой.

## Прогон 6 — события HID_EVENT20 (`probe-events.ps1`)

✅✅ **События ЛОВЯТСЯ из пользовательской сессии** (подписка на `HID_EVENT20` сработала). Только класс `HID_EVENT20` существует (10/00/30 — нет).

Формат: `EventDetail[0]=0x01` (тип=HID), `[1]` = код клавиши, `[2]` = значение.

| код `[1]` | знач `[2]` | предположение |
|-----------|-----------|---------------|
| `0x21` | 0/1 | микрофон mute (подтв. MIControl `HandleFnKey`) |
| `0x05` | 0…5 | яркость подсветки клавиатуры (уровень в `[2]`) |
| `0x25`/`0x26` | 1 | **Mi-кнопка** (0x25=нажатие, 0x26=отпускание). В коде реагируем только на 0x25. |
| `0x23`/`0x24` | 1 | пара — клавиша press/release |
| `0x1B` | 0 | Mi-помощник? |
| `0x01` | 0 | Mi-кнопка / монитор? |

- Нажаты вперемешку: микрофон, отключение монитора, Mi-помощник, подсветка, Mi-кнопка.
- Точные ярлыки → добить одиночным прогоном (по одной клавише) на этапе OSD/hotkeys (v0.3).

### 🏁 АРХИТЕКТУРНЫЙ ИТОГ
**Служба НЕ нужна.** Всё из трей-резидента (admin): SET/GET функций ✓, подписка на события ✓. → **План А** (см. `03-architecture.md`).

## Прогон 5 — режимы производительности (`probe-perf.ps1`)

Текущий режим: **Auto (0x09)**. Результат SET+readback:

| arg | Режим | На TM2424 |
|-----|-------|-----------|
| `0x01` | Balance | ❌ **не принят** (остаётся Auto) — как в MIControl для TM24 |
| `0x02` | Quiet | ✅ принят |
| `0x03` | Turbo | ✅ принят |
| `0x04` | Full-speed | ✅ принят |
| `0x09` | Auto | ✅ принят |

- ✅ **SET режима работает** (обычный admin). GET → режим в `OUT[4]`.
- Доступно **4 режима**: Тихий / Турбо / Full-speed / Авто (без Balance).
- Для UI: Тихий(0x02) / Авто(0x09) / Турбо(0x03) (+ опц. Full-speed 0x04).
- (баг скрипта `Write-Host -f` не помешал: последний SET в цикле = Auto = исходный.)

## Прогон 4 — эксперимент «в бою» (`charge.ps1`)

### `-Limit 60` (батарея 56%)
- Защита `0x02`: 0 → **1** ✅ (включилась)
- Лимит `0x01`: остался **100** — запись «60» НЕ принята (повтор результата probe-set). Под-функция `0x01` не пишется этим путём. Устойчиво.
- Флаг `0x03`: постоянно **1** (и на батарее, и при включённой защите) — вероятно «capability/поддерживается».

### Наблюдение зарядки (подключён БП) — В ПРОЦЕССЕ

| Момент | Заряд | 0x01 | 0x02 | 0x03 |
|--------|-------|------|------|------|
| подключение | 56% | 100 | 1 | 1 |
| — | 60%+ (пролетел) | 100 | 1 | 1 |
| — | 70% | 100 | 1 | **0** |
| — | 71% | 100 | 1 | 0 |
| — | 72% (**всё ещё растёт**) | 100 | 1 | 0 |

### ⚠️ ПОСПЕШНЫЙ ВЫВОД ОТМЕНЁН
Ранее ошибочно заключил «встал на 70%» по одному замеру. На деле **заряд продолжился** (71, 72, …). Значит:
- Включение `0x02`=1 (с нашим ре-армом) **НЕ ограничило заряд на 70%** на этой модели.
- `0x03` флипнул 1→0 около 70%, но заряд идёт → **`0x03` ≠ «идёт заряд»**. Смысл неизвестен.
- Модель заряда **НЕ определена**. Ждём потолок (80/90/100?).

### Ведущая гипотеза: сброс при смене питания
MIControl (`CMIService_Client.cpp:104`) на кнопку заряда зовёт `guard->PreserveStatus()`, а guard (`CPowerPowerChargeProtectGuard`) **переустанавливает `Put(0x10,0x02,0→1)` при КАЖДОЙ смене питания и выходе из сна** — потому что EC сбрасывает защиту на этих событиях. Механизм: только вкл/выкл, БЕЗ процента (порог фиксирован прошивкой).

Мы включили защиту **на батарее**, затем воткнули БП → смена питания, вероятно, **сбросила** enforcement (регистр `0x02` показывает 1, но EC уже сброшен) → заряд идёт свободно.

### Проверка (charge.ps1 -Rearm на AC) — тоже НЕ ограничил
Переарм `0x02` off→on на AC: 74% → 77% и **дальше растёт**. Заряд не остановился. Идёт к 100.

### Заряд ВСТАЛ на 80% + Windows показал «оптимизированная зарядка»
Заряд остановился на **80%**, иконка стала «от сети / не заряжается», Windows вывел уведомление **«с помощью функции оптимизированная зарядка»**.

⚠️ **Кто держит 80% — НЕ ясно:**
- наш WMI (`0x02`=1) с порогом 80, ЛИБО
- **Windows Smart Charging** (собственная функция Win11, держит ~80% на сети) — уведомление именно её формулировкой.

### Тест на разведение (charge.ps1 -Off) — РЕЗУЛЬТАТ
- Удержание на 79–80%: `0x01`=100, `0x02`=1, `0x03`=0.
- **`-Off` (0x02→0) → заряд ПОШЁЛ дальше. Снова включить (0x02→1) → снова встаёт.** Повторяемо.

### ✅✅ ИТОГ ПО ЗАРЯДУ — ПОДТВЕРЖДЁН WMI-МЕХАНИЗМ
- **Заряд управляется нашим WMI**, не Windows (доказано переключением: off→заряжает, on→стоп).
- Команды: **`SET 0x10 sub=0x02 val=1`** = беречь (стоп ~80%); **`val=0`** = заряжать до 100%.
- Статус лимита: `GET 0x10 sub=0x02` → OUT[6] (0/1).
- **Порог фиксирован прошивкой ≈ 80%** (не 70; раннее «70» — ошибка чтения). `0x01`=100 на отсечку не влияет, произвольный % недоступен.
- Windows-уведомление «оптимизированная зарядка» = просто отражение EC-удержания, не причина.
- `0x03`: 1→0 около порога; точный смысл не критичен (индикатор зоны заряда).

### Модель для приложения
**Тумблер:** «Беречь батарею» (≈80%) ↔ «Заряжать полностью» (100%). Бинарно, как и договаривались.
Открыто: сохраняется ли `0x02`=1 после сна/смены питания (нужен ли ChargeGuard) — проверить отдельно.
